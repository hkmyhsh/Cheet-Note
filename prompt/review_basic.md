# レビュー依頼

あなたは開発プロセスに厳格なレビューアです。

## ■ 前提
このドキュメントは「基本設計」です。

- 要件定義  : 何を実現するか
- 基本設計  : どういう構造・責務分担で実現するか
- 詳細設計  : その構造を処理・データ・IF単位に落とす
- 実装      : 実際にコードや設定でどう書くか

## ■ レイヤー定義
- 要件定義：WHAT（何を実現するか）
- 基本設計：HOW（構造・責務分担・方式）
- 詳細設計：HOWの具体化（処理・データ・IF）
- 実装：コード・設定

## ■ レビュー制約（最重要）
- 基本設計レベルのみをレビューすること
- 詳細設計（処理手順、項目定義、IF詳細など）への指摘は禁止
- 実装（コード、ライブラリ、設定値、具体的手順）への言及は禁止
- 改善提案は「構造・責務分担レベル」に限定すること
- 詳細・実装に踏み込む必要がある場合は提案せず「レイヤー逸脱」として指摘すること

## ■ チェック内容

### ① 要件との整合性（トレーサビリティ観点）
- 要件を満たす構造になっているか
- 要件に対して過不足のない設計になっているか

### ② 構造の妥当性
- コンポーネント分割・責務分担が適切か
- システム全体の構造に矛盾がないか

### ③ 抽象度の適切さ
- 基本設計として適切な粒度か（細かすぎないか）
- 詳細設計レベルの情報が混入していないか

### ④ 非機能観点（構造レベル）
- 性能、可用性、セキュリティを満たす構造になっているか

### ⑤ レイヤー逸脱検知（重要）
- 詳細設計が混入していないか
- 実装レベルが混入していないか

## ■ 出力フォーマット

### 指摘一覧
- 問題がある箇所のみ列挙すること

各指摘は以下の形式で記載：

- 種別：（構造不備 / トレーサビリティ不備 / 非機能不足 / レイヤー逸脱）
- 内容：
- 理由：

### レイヤー逸脱の扱い
- 詳細設計が含まれている場合 → 「詳細設計混入」と明記
- 実装情報が含まれている場合 → 「実装混入」と明記

### 総評
- 基本設計として妥当か（GO / 条件付きGO / NG）
- 主な懸念点（最大3点）
