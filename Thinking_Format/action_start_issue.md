# 大前提
## 自分の特性上
### 得意なこと
- 高次の理解・抽象化・構造化は非常に得意
- 「材料が明示されている状態」での設計・改善は強い
### 絶対的に苦手なこと
- 曖昧
- 口頭のみ
- 暗黙前提が多い
- 即時判断が求められる
という状況では処理が詰まりやすい

# 仕事をする上で
## 上司の期待値
- 依頼文の裏にある「背景・目的」を推定してほしい
- 自分（上司）が考える「妥当なゴール」を察してほしい
- 忙しいため、説明コストを省きたい
つまり、上司は
「依頼 → いい感じに補完してやっておいてほしい」
という暗黙補完型の仕事の振り方をしています。

## 現状の課題
- 自分には再現性がない
  - 推測がうまく当たったときだけ成功している
  - 成功要因が言語化・固定化されていない
  - 調子・疲労・情報量に左右される
  - 「暗黙知の推定」は確率論的処理
  - 疲労や情報ノイズで成功率が大きく上下

## 課題の解決
- 【最重要】「察する」ことを仕事から排除し、「確認する」ことを仕事に組み込む
  - **「考える部下」ではなく「整理して判断させる部下」**になる
    - サービスマネジメント / 運用改善エンジニア としての立ち位置
- 明文化・確認・型化　による安定運用

### 課題解決の思想
- 「あなたが安定して受け取れること」 と 「上司が最小コストで判断できること」 
  - 察しない
  - 仮説を置く
  - 判断だけを依頼する
  - 後から「言った・言わない」が起きない

### 依頼を受けたら必ず「再構成して返す」

```
・目的：
・背景：
・期待するアウトプット：
・粒度（調査／設計／実装／提案）：
・期限：
・判断が必要な点：
この理解で合っていますか？
```

#### 「仮のゴール」を明示して確認する

```
仮に以下をゴールと考えています。

・成果物形式：〇〇（資料／手順書／改善案）
・対象範囲：〇〇まで
・深さ：運用担当が再現可能なレベル

ズレていたら修正ください。
```
