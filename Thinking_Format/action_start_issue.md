# 大前提
## 自分の特性上
### 得意なこと
- 高次の理解・抽象化・構造化は非常に得意
- 「材料が明示されている状態」での設計・改善は強い
### 絶対的に苦手なこと
- 曖昧
- 口頭のみ
- 暗黙前提が多い
- 即時判断が求められる
という状況では処理が詰まりやすい

# 仕事をする上で
## 上司の期待値
- 依頼文の裏にある「背景・目的」を推定してほしい
- 自分（上司）が考える「妥当なゴール」を察してほしい
- 忙しいため、説明コストを省きたい
つまり、上司は
「依頼 → いい感じに補完してやっておいてほしい」
という暗黙補完型の仕事の振り方をしています。

## 現状の課題
- 自分には再現性がない
  - 推測がうまく当たったときだけ成功している
  - 成功要因が言語化・固定化されていない
  - 調子・疲労・情報量に左右される
  - 「暗黙知の推定」は確率論的処理
  - 疲労や情報ノイズで成功率が大きく上下

## 課題の解決
- 【最重要】「察する」ことを仕事から排除し、「確認する」ことを仕事に組み込む
  - **「考える部下」ではなく「整理して判断させる部下」**になる
    - サービスマネジメント / 運用改善エンジニア としての立ち位置
- 明文化・確認・型化　による安定運用

### 課題解決の思想
- 「あなたが安定して受け取れること」 と 「上司が最小コストで判断できること」 
  - 察しない
  - 仮説を置く
  - 判断だけを依頼する
  - 後から「言った・言わない」が起きない

### 依頼を受けたら必ず「再構成して返す」

```
・目的：
・背景：
・期待するアウトプット：
・粒度（調査／設計／実装／提案）：
・期限：
・判断が必要な点：
この理解で合っていますか？
```

#### 「仮のゴール」を明示して確認する

```
仮に以下をゴールと考えています。

・成果物形式：〇〇（資料／手順書／改善案）
・対象範囲：〇〇まで
・深さ：運用担当が再現可能なレベル

ズレていたら修正ください。
```

## まとめ：材料チェックリスト（簡易）
- 作業開始前に、最低限これが言語化できていれば着手OKです。
  - 目的と完了条件は明確か
  - 対象範囲と制約は分かっているか
  - 現状情報は手元にあるか
  - 判断基準は定義されているか
  - 必要な権限・人・ツールは揃っているか
  - 出力形式は決まっているか

# テンプレートサンプル
## 使い方
- 最初から全部埋めなくてよい
- 2〜3項目埋めて「この理解で合っていますか？」と返す
- 上司の返答で空欄を埋める
- 合意後に作業開始
```
これは
「察する部下」→「整理して判断させる部下」 への移行装置です。
```

## ① 運用改善依頼 受け取りテンプレ（A4・1枚）
```
1. 依頼の要点（まずここだけ埋める）
	•	依頼タイトル：
	•	対象システム／業務：
	•	発生している課題（現象ベース）：
	•	例：作業が属人化している／手順が古い／工数が読めない 等

⸻

2. 背景・目的（推測含む）
	•	背景（なぜ今この話が出たか）：
	•	最終的に達成したい状態（目的）：
	•	例：引き継ぎ可能にしたい／工数削減したい／事故を減らしたい

※ここは仮説でOK。後述の確認で合意する。

⸻

3. 想定アウトプット（仮置き）
	•	成果物の形式：
	•	☐ 改善案資料
	•	☐ 手順書
	•	☐ フロー図
	•	☐ 実装（スクリプト／設定変更）
	•	粒度：
	•	☐ 概要提案レベル
	•	☐ 実運用でそのまま使えるレベル
	•	対応範囲：
	•	（例：本番作業まで／提案まで）

⸻

4. 制約・前提条件
	•	期限：
	•	触ってよい／よくない範囲：
	•	関係者（レビュー・合意が必要な人）：

⸻

5. 判断が必要な点（上司向け）
	•	上記の目的理解は合っていますか
	•	アウトプットの粒度はこれで良いですか
	•	範囲を広げる／狭める必要はありますか

⸻

6. 合意事項（確定欄）
	•	確定した目的：
	•	確定した成果物：
	•	次アクション：
```

## ② 調査依頼 受け取りテンプレ（A4・1枚）
```
1. 調査対象の明確化
	•	調査テーマ：
	•	対象環境／システム：
	•	発生している疑問・不安：

⸻

2. 調査の目的（重要）
	•	この調査で何を判断したいか：
	•	例：採用可否判断／原因切り分け／影響範囲把握

※「調べること」ではなく「判断」を明示する。

⸻

3. 調査範囲（仮置き）
	•	含める範囲：
	•	含めない範囲：
	•	例：実装検証は含めない／机上調査のみ 等

⸻

4. 想定アウトプット
	•	成果物形式：
	•	☐ 調査結果まとめ
	•	☐ 比較表
	•	☐ 推奨案＋理由
	•	結論の有無：
	•	☐ 事実整理のみ
	•	☐ 推奨案まで出す

⸻

5. 判断ポイント（上司向け）
	•	どこまで結論を出す必要がありますか
	•	想定と異なる結果が出た場合の扱い
	•	☐ 追加調査
	•	☐ ここで打ち切り

⸻

6. 合意事項
	•	調査ゴール：
	•	成果物：
	•	期限：
```

## ③ 障害対応依頼 受け取りテンプレ（A4・1枚）
```
1. 事象の事実整理（確定情報）
	•	発生日時：
	•	影響範囲：
	•	システム：
	•	ユーザー：
	•	現在の状態：
	•	☐ 継続中
	•	☐ 暫定復旧
	•	☐ 完全復旧

⸻

2. 期待されている役割（重要）
	•	自分に期待されていること：
	•	☐ 切り分け
	•	☐ 原因特定
	•	☐ 復旧対応
	•	☐ 再発防止案
	•	☐ 報告資料作成

※複数選択可。ここを必ず明示。

⸻

3. 優先度・時間軸
	•	優先度：
	•	☐ 最優先（他作業中断）
	•	☐ 並行対応
	•	期限感：
	•	いつまでに何が必要か

⸻

4. 判断・エスカレーション条件
	•	どの時点で報告／相談するか：
	•	勝手に決めてよい範囲：
	•	上司判断が必要な範囲：

⸻

5. 想定アウトプット
	•	即時：状況共有（口頭／チャット）
	•	事後：
	•	☐ 障害報告書
	•	☐ 再発防止策
	•	☐ 運用改善提案

⸻

6. 合意事項
	•	今回のゴール：
	•	対応範囲：
	•	次アクション：
```
