# ミニサマリ → 要件定義・設計書への橋渡し

## 橋渡しの手法の目的
- 下記の通りの分類ができていること
  - ❌ 考えながら設計書を書く
  - ⭕ 考え終わったものだけ設計書に置く
  - 各ポイントの役割
    - 思考ダンプ
      - 考えるための場所
    - ミニサマリ
      - 止めるための場所
    - 要件・設計
      - 決まったことを書く場所

## 橋渡しの最重要ポイント
- ミニサマリは 要件定義・設計書そのものにしない ことが最大のコツです。
  - ❌ ミニサマリを清書しようとする
  - ⭕ ミニサマリを「仕分け」して、適切な箱に移す

## ミニサマリの定義
- 下記3点の混在物
  - A. 確定した事実（もう決まっている）
  - B. 判断途中・未決事項
  - C. 次に調べる／考える作業

## 橋渡しとは

### 作業全体フロー
- *文章は書かない。分類だけ。*
```
ミニサマリ
↓（分類する）
① 事実
② 判断が必要
③ 作業タスク
↓
要件定義 / 基本設計 / 詳細設計 に振り分け
```

### ミニサマリ → 分類の具体手順（5〜10分）
```
【作業名】
EKS Ingress HTTPS化（社内CA）

【どこまでやったか】
ALBでのHTTPS終端設定まで完了

【今やめた理由】
社内CA証明書のPod配布方法で詰まった

【次にやる1アクション】
Podへの証明書配布パターンを整理する
```

## Step1：（頭の中でOK）色分け or ラベル付け
- 🟢 事実
  - ALBでHTTPS終端設定まで完了
- 🟡 判断が必要
  - 社内CA証明書のPod配布方法
- 🔵 作業タスク
  - 配布パターンを整理する

###（超重要）仕分けルール
- *機会的なルールに従う*

### 🟢 事実 → 要件 or 設計の「決定事項」
- 条件
  - すでに実装 or 合意済み
  - 後戻りしない前提
- 行き先
  - 要件定義：スコープ／制約
  - 基本設計：構成・方針
  - 詳細設計：処理・設定

#### 例
```
【要件定義】制約事項
- HTTPS終端はALBで行う
```

### 🟡 判断が必要 → 未決事項・レビュー項目
- 条件
  - 「どうするか」が決まっていない
  - 複数案がありそう
- 行き先
  - 要件定義：未決事項
  - 基本設計：検討事項
  - レビューコメントの元ネタ

#### 例
```
【要件定義】未決事項
- 社内CA証明書をPodへ配布する方式
```

### 🔵 作業タスク → 調査・TODO（成果物に書かない）
- 条件
  - 調べる／試す／比較する
- 行き先
  - 自分のタスクメモ
  - Issue / TODO
  - 次回の思考ダンプ
- ❌ 要件定義・設計書には書かない

## 要件定義・設計書への「最低限の昇格」
- *ここで初めての文章化*

### 要件定義への昇格（最小）
- *背景説明は不要*
- ミニサマリが裏にあるので問題なし
```
## 制約事項
- HTTPS通信はALBで終端する

## 未決事項
- 社内CA証明書の配布方式
```

### 基本設計への昇格（最小）
```
## セキュリティ設計方針
- TLS終端：ALB
- Pod間通信：未定（検討中）
```
