# ミニサマリ → 要件定義・設計書への橋渡し

## 最重要
- ミニサマリは 要件定義・設計書そのものにしない ことが最大のコツです。
  - ❌ ミニサマリを清書しようとする
  - ⭕ ミニサマリを「仕分け」して、適切な箱に移す

## ミニサマリの定義
- 下記3点の混在物
  - A. 確定した事実（もう決まっている）
  - B. 判断途中・未決事項
  - C. 次に調べる／考える作業

## 橋渡しとは

### 作業全体フロー
- *文章は書かない。分類だけ。*
```
ミニサマリ
↓（分類する）
① 事実
② 判断が必要
③ 作業タスク
↓
要件定義 / 基本設計 / 詳細設計 に振り分け
```

### ミニサマリ → 分類の具体手順（5〜10分）
```
【作業名】
EKS Ingress HTTPS化（社内CA）

【どこまでやったか】
ALBでのHTTPS終端設定まで完了

【今やめた理由】
社内CA証明書のPod配布方法で詰まった

【次にやる1アクション】
Podへの証明書配布パターンを整理する
```

### Step1：（頭の中でOK）色分け or ラベル付け
- 🟢 事実
  - ALBでHTTPS終端設定まで完了
- 🟡 判断が必要
  - 社内CA証明書のPod配布方法
- 🔵 作業タスク
  - 配布パターンを整理する

##（超重要）仕分けルール
- *機会的なルールに従う*

### 🟢 事実 → 要件 or 設計の「決定事項」
- 条件
  - すでに実装 or 合意済み
  - 後戻りしない前提
- 行き先
  - 要件定義：スコープ／制約
  - 基本設計：構成・方針
  - 詳細設計：処理・設定
