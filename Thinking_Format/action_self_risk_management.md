# チームのブレーキにならないために

## 自分の傾向

	•	細部の不整合が気になる
	•	リスクの芽が見える
	•	曖昧なまま進むことに違和感を覚える
	•	Mtgの結果が「具体行動」まで落ちないと落ち着かない

 → **「未完了状態」に対する耐性が低い**

### 傾向的にはABS（制御付きブレーキ）になれる

下記を防ぐ役割

	•	属人化
	•	暗黙判断
	•	障害多発
	•	再発

ただし、**止めるべき曖昧さ / 流して良い曖昧さ**を分けないとただのブレーキになってしまう

### ブレーキになりやすいパターン

	•	問題提起が長い
	•	代替案がない
	•	今やるべきかどうかの整理がない
	•	相手の時間軸を無視する

## 会議などでの本質的なずれ

今の構造は：

	•	自分：事故前提で考える
	•	上司：進行前提で考える

つまり

時間軸が違う。

## 回避 or 改善策

### リスクに優先度タグをつける

	•	P1: 今止めないと事故
	•	P2: 次スプリントで整理
	•	P3: 将来負債

### 会議中の発言

**発言は最大1行。**

会議で発言する前に自分に問う：

	•	レベル1：確認質問で済むか？
	•	レベル2：懸念を1行で言えるか？
	•	レベル3：構造説明が必要か？

**原則：レベル1か2で止める。**

### 会議では「止める」より「条件付きで進める」

```
×「このままでは危険です」
○「この条件が満たせるなら進めましょう」
```

### 会議後処理モデル

 1.	懸念ログを3行で整理
	2.	上司に1つだけ共有
	3.	あとは手放す

**全部抱えない。**

## 再定義

「事故を防ぐために議論を止める」ではなく、**「事故を防ぐ“仕組み”を入れること」**

```
ブレーキになるのは、

“止める”から。

でも本質的にすべきことは

“未来の事故を構造化できること”。

今はタイミングが会議内になっている。

それを「会議外」に移す。

```

### 具体的な転換

【転換前】目の前で潰す

→ あなたが疲れる
→ ブレーキに見える

【転換後】構造に吸収させる

→ あなたは静かになる
→ チームは安定する

```
今 → これから
会議で止める → 会議の外で仕組みにする
指摘する → 形式知化する
その場で詰める → 再現可能なチェックにする
```

#### 会議中に解決しない

気づく → その場で解決しようとする

から

**気づく → 記録する → 後で構造化（処理）する**

#### 「リスクログ」を制度化する

一人で抱えない。

例：
「懸念はここに集約します」
「スプリント後半でレビューします」

#### 転換モデル

Step 1: 会議中は “タグ付けだけ” する

心の中で：
	•	[今止める]
	•	[後で検証]
	•	[ログだけ残す]

と分類する。

口に出さない。

⸻

Step 2: 会議後に「構造メモ」を作る

3行でいい。

例：
	•	前提Aが曖昧
	•	例外系未定義
	•	運用責任者未確定

これだけ。

⸻

Step 3: 週1でまとめて出す

会議で都度止めない。

「今週の懸念まとめ」として出す。

これなら進行は止まらない。
