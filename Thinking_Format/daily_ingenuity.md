# ① 仕事（ITエンジニア・設計・運用）での具体的な工夫

## 1. 設計フェーズでの工夫

### 「曖昧さ」を前倒しで潰す
- 「後工程の曖昧さ」が一番のストレス源になりやすいです。
- 実践ポイント
  - 要件は必ず以下を明示
    - 目的（Why）
    - ゴール（Doneの定義）
    - 制約（やってはいけないこと）
  - 「想定外が起きたらどうするか」を設計に含める
  - 例外パターンを先に洗い出す
  - 👉設計レビューで価値が最大化される役割です。

### 図と文章をセットで残す
- Mermaid / draw.io / C4 などで構造図を先に描く
- 文章は「図の補足説明」として書く
- 👉「考えを外に出す」ことで、思考負荷と疲労が激減します。

## 2. 運用フェーズでの工夫

### 割り込みを減らす仕組みを作る
- 口頭相談 → チケット／Issue化
- 緊急度ラベルを明示
- 即レス期待を下げる（SLAを決める）
- 👉集中力＝資源として扱うのが重要です。

### 手順は「判断不要」に寄せる
- 「考えて対応」ではなく「読んで実行」
- if / then を明文化
- 判断が必要な場合は判断基準を明記
```
× 状況に応じて対応
○ CPU使用率80%以上が5分継続したらAを実施
```

## コミュニケーションの工夫（かなり重要）
### 構造で話すテンプレ
- 感情や雰囲気ではなく、構造で話すと誤解が減ります。
  - 「冷たい」ではなく「整理されている人」
- おすすめフォーマット
```
① 事実（観測）
② 問題（構造）
③ 影響範囲
④ 提案（選択肢）
```

### 「考える時間」を明示する
- 即答がつらい場面では：「一度整理してから回答します。今日中にまとめます」
- これは責任放棄ではなく品質担保

### 疲労・体調管理
- 集中できる時間帯を把握する
- 午前／午後で役割を分ける
- 感覚過負荷（音・会話・通知）を減らす
- 「頑張れる日」を基準にしない
  - →「低調日でも回る設計」にする
