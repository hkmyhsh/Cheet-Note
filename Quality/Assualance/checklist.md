# 開発品質保証チェックリスト（軽量・実務向け）

## 目的
最低限これだけ守れば品質事故率を大幅に下げるための、  
**軽量・低負荷・工程別チェックリスト**。

---

# 全体方針

## 設計原則
- 各工程 5〜10項目以内
- Yes/Noで判断可能
- レビュー観点を統一
- 記載負荷を最小化
- 工程責務ごとに分離
- 品質を属人化させない

---

# ① 要件定義チェックリスト

## パターンA：標準型
| No | チェック項目 |
|----|----------------|
| 1 | 実現したい目的・背景が明確か |
| 2 | 対象ユーザー/利用者が定義されているか |
| 3 | 機能要件が具体的か |
| 4 | 非機能要件（性能/可用性/セキュリティ）があるか |
| 5 | 制約条件（期限・環境・依存）が明記されているか |
| 6 | 成功条件（完了条件/KPI）が定義されているか |
| 7 | 想定リスクが整理されているか |
| 8 | スコープ外が定義されているか |

## パターンB：超軽量型
- [ ] 何を作るか
- [ ] なぜ必要か
- [ ] 誰が使うか
- [ ] 成功条件は何か
- [ ] 主なリスクは何か

# 推奨運用方法

## 初期導入
- 各工程5項目版導入
- テンプレート統一
- 強制レビュー

## 安定後
- 8項目版へ拡張
- トレーサビリティ表連携
- リスク分類表統合
- Runbook統合

---

# サボり防止ポイント
- 項目数を少なく保つ
- 必須項目のみ
- Yes/No判定
- PR前提に組込
- Markdownテンプレ固定
- レビュー文化ではなく工程ルール化

---
