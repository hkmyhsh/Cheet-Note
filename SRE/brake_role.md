
# 役割

**「止める人」ではなく「減速を設計する人」**
👉 **「事故確率を下げる制御装置」**

## 避けるべき急ブレーキの典型パターン

- リスクが「見えていなかった」→直前で顕在化
- 判断材料が揃うのが遅い
- 「違和感」はあったが言語化されていない
- チームが速度優先で観測を省略している

👉 **検知が遅い = 強いブレーキになる**

## 理想状態

**“止める人”ではなく「速度と安全のバランスを制御する人」**

	•	NG：閾値を超えたら停止
	•	OK：閾値に近づく前に減速

# ブレーキ役の動き方

## ブレーキのかけ方

### 技術1：止めない問い（重要）

❌ 悪いブレーキ

	•	「これ危なくないですか？」

✅ 良い減速

	•	「この前提が外れた場合の影響だけ先に見ておきたいです」

👉 ポイント：

	•	止めない
	•	観測を増やす

### 技術2：範囲限定ブレーキ

❌ 全体停止

	•	「一旦全部見直しましょう」

✅ 部分減速

	•	「この変更だけリスク切り出して見たいです」

👉 チームの体感：

	•	止められた → ストレス
	•	限定された → 納得

### 技術3：時間制限付き減速

	•	「10分だけリスク洗い出しさせてください」
	•	「ここだけ確認してから進めたいです」

👉 無限に考えない構造を作る

### 技術4：先回りの“予防ブレーキ”

「軽くブレーキ」はこれ

タスク開始時に

	•	前提
	•	依存
	•	影響範囲

を軽く確認する

👉 これをやると
後半の急ブレーキが消える

##ブレーキの強さ調整ルール

| ブレーキの予兆 | ブレーキ |
| ---------- | ------ |
| 情報欠損あり | 軽く減速（質問） |
| 不可逆変更あり | 中ブレーキ（確認） |
| 影響大 + 不明確 | 強ブレーキ（止める） |

👉 重要：
**全部を止めないこと**

### 「ブレーキの予兆」＝減速トリガー

#### 情報の欠損（最も重要）

	•	要件の前提が曖昧
	•	「たぶん」「仮」で進んでいる
	•	依存関係が不明確

👉 これは事故の最大因子

#### 不可逆性の増大

	•	後戻りコストが高い変更
	•	データ変更、権限変更、公開API
	•	一度やると戻せない

👉 ここで減速しないと急ブレーキ確定

#### 認識のズレ

	•	レビューコメントがバラつく
	•	同じ言葉で違う意味を使っている
	•	「それって○○前提ですよね？」が出る

👉 ズレは後半で爆発する
