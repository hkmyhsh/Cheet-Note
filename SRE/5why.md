# SREにおける問題解決の全体像

SREの体系的アプローチは、概ね以下の流れで構成されます。

```
① 検知
② 影響評価
③ 緊急対応（Mitigation）
④ 根本原因分析（RCA）
⑤ 再発防止設計
⑥ 仕組みへの組み込み
⑦ 学習と改善
```

## SREで言う体系的とは

```
✔ 属人性を排除する
✔ 再現可能である
✔ 抽象度を上げる
✔ 設計へ還元する
✔ 数値（SLO）で回す
```

という意味

## Googleが提唱するSREの基本思想
	•	エラーバジェット
	•	トイル削減
	•	ポストモーテム文化
	•	SLO駆動設計


# SREの問題解決は「設計変更」で終わる

## SREではこう考えます：

### インシデントは“バグ”ではなく、“設計のフィードバック”

つまり

	•	手順を増やす → ×
	•	人に注意する → ×
	•	自動化する → ○
	•	制御点を追加する → ◎
	•	構造を変更する → ◎◎

## 例

なぜ503が出た？
→ スレッドが枯渇した

なぜ枯渇した？
→ リトライが暴走した

なぜ暴走した？
→ タイムアウト設計がなかった

なぜ設計しなかった？
→ 非機能要件に明示していなかった

なぜ日機能要件に明示しなかったか
→ 非機能要件レビュー工程が存在しない(ここまで行きたい)

# SRE型ミニ問題解決テンプレ

① 現象

何が起きた？

② 直接原因

技術的に何が壊れた？

③ 構造原因

なぜ設計で防げなかった？

④ 再発防止の種類は？

	•	設計変更？
	•	監視追加？
	•	自動化？
	•	テスト追加？

⑤ 再現性は上がったか？

同じ事故は“理論上”起こるか？


# SRE型制御点テンプレ

Step1：現象

何が起きた？

Step2：増幅経路

どうやって悪化した？

Step3：制御点欠如

どこに制御点が無かった？
	•	制限？
	•	観測？
	•	反応？

Step4：追加すべき制御点

何を追加する？

Step5：再発可能性評価

理論上、同じ事故は起きるか？
