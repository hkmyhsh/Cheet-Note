# 減速チェックリスト（軽量版 / 5〜10分）

	•	毎回全部チェックしない
	•	違和感がある箇所だけ使う

## 使う際の一言

	•	「ここだけ軽く見たいです」
	•	「この前提だけ揃えたいです」
	•	「戻し方だけ確認したいです」

👉 部分減速 + 言語化



## 0. 実行ルール
- ❌ 止めるために使わない
- ✅ 観測と前提確認のために使う
- ⏱️ 最大10分（長引かせない）

---

## 自分用制御
- [ ] 10分以上考えていないか？
- [ ] 全体を止めようとしていないか？
- [ ] 「一部だけ」に絞れているか？

---

## 1. 前提の確認（情報欠損の検知）
- [ ] 「何をやるか」を1行で説明できる
- [ ] 対象範囲（どこまで影響するか）が明確
- [ ] 除外範囲（やらないこと）が明確
- [ ] 「仮」や「たぶん」が含まれていないか

👉 1つでも曖昧なら：
「前提だけ軽く揃えたいです」と減速

---

## 2. 依存関係の確認（見えないリスク）
- [ ] 外部サービス依存（GitHub / AWS / Nexus 等）がある
- [ ] 他チーム・他PJへの影響がある
- [ ] 実行順序に依存がある（順番ミスで壊れる）

👉 1つでもあるなら：
「依存だけ切り出して見たいです」と減速

---

## 3. 不可逆性チェック（急ブレーキ防止）
- [ ] データ変更（削除・上書き）がある
- [ ] 権限変更（IAM / 認証）がある
- [ ] 公開範囲変更（外部公開・URL変更）がある
- [ ] ロールバック手順が未定義

👉 YESなら：
「戻し方だけ先に決めたいです」と中ブレーキ

---

## 4. 認識ズレチェック（後半爆発防止）
- [ ] 用語の定義が人によって違っていない
- [ ] 成功条件が一致している
- [ ] レビュー観点が共有されている

👉 違和感があれば：
「成功条件だけ揃えたいです」と減速

---

## 5. 影響範囲の可視化（事故範囲の限定）
- [ ] 影響ユーザーが特定できている
- [ ] 影響時間帯（いつ影響するか）が分かる
- [ ] 監視・検知方法がある

👉 不明なら：
「影響範囲だけ見たいです」と減速

---

## 6. 実行・検証の最小単位化
- [ ] 小さく試せる（いきなり本番でない）
- [ ] 段階的にリリースできる
- [ ] 成功/失敗の判断がその場でできる

👉 できない場合：
「一段階分割できないか見たいです」と減速

---

## 7. 終了条件（止めすぎ防止）
- [ ] どこまで確認したら進めるか決まっている
- [ ] 完璧を求めていない（80%で進む）
